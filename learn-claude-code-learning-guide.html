核心洞察：agent_loop 永远不变

整个项目最重要的设计决策：核心循环的代码从 s01 到 s12 结构完全不变。每一课只是在循环之上叠加一层新机制（更多工具、注入提醒、后台线程……），循环本身始终是同一个 while True。

下面展示一次标准循环代码。后续章节将不再重复这个循环，只关注每课新增的部分。

def agent_loop(messages):
    while True:
        # 1. 把对话历史发给 LLM，让它思考下一步
        response = client.messages.create(
            model=MODEL, system=SYSTEM,
            messages=messages, tools=TOOLS,
        )
        # 2. 把 LLM 的回复追加到对话历史
        messages.append({"role": "assistant", "content": response.content})

        # 3. 如果 LLM 没有要求调用工具 → 任务完成，退出循环
        if response.stop_reason != "tool_use":
            return

        # 4. LLM 要求调用工具 → 执行每个工具调用，收集结果
        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = TOOL_HANDLERS[block.name](**block.input)
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        # 5. 把工具结果以 "user" 角色追加 → 回到步骤 1
        messages.append({"role": "user", "content": results})

核心循环模式解析

想象一个永不下班的实习生。老板给他一个任务，他就开始干。如果需要查资料，他就去查（调用工具）；查完了把结果拿回来继续干。直到他觉得"做完了"，才停下来汇报。这个 "接任务 → 需要工具就用 → 结果拿回来 → 继续 → 做完才停" 的循环，就是 Agent 的全部秘密。

流程图

    |   User   | ---> |  LLM  | ---> |  Tool   |
    |  prompt  |      |       |      | execute |
    +----------+      +---+---+      +----+----+
                         ^               |
                         |   tool_result |
                         +---------------+
                         (loop continues)

# 第 67 行 — 函数签名：接收一个 messages 列表（对话历史）
def agent_loop(messages: list):
    while True:                        # 无限循环，直到 LLM 决定停止
        response = client.messages.create(  # 调用 Anthropic API
            model=MODEL,                    # 使用哪个模型
            system=SYSTEM,                  # 系统提示词（角色设定）
            messages=messages,              # 完整对话历史
            tools=TOOLS,                    # 可用工具列表
            max_tokens=8000,                # 最多生成 8000 token
        )
        # 把 LLM 的回复追加到对话历史（assistant 角色）
        messages.append({"role": "assistant", "content": response.content})

        # 关键判断：LLM 是否要求调用工具？
        if response.stop_reason != "tool_use":
            return             # 不需要工具 → 任务完成，退出

        # LLM 要求调用工具 → 逐个执行
        results = []
        for block in response.content:
            if block.type == "tool_use":
                print(f"$ {block.input['command']}")   # 打印命令
                output = run_bash(block.input["command"])  # 执行
                print(output[:200])                         # 打印结果
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,   # 关联到哪个工具调用
                    "content": output,          # 工具输出
                })
        # 把工具结果以 "user" 角色追加，然后回到 while True 顶部
        messages.append({"role": "user", "content": results})

关键要点

  • messages 列表 = Agent 的全部记忆。每次循环都把 LLM 回复和工具结果追加进去。
  • LLM 本身无状态——它不记得上一次调用。所有记忆都存在 messages 里，每次调用都完整发送。
  • stop_reason 是唯一的退出信号：LLM 觉得做完了就不再调用工具，循环自然结束。

s01 — Agent Loop

"One loop & Bash is all you need" — 一个工具 + 一个循环 = 一个智能体
s01_agent_loop.py
107 LOC 1 工具 起点

这是整个课程的起点。一个 Python 文件，一个工具（bash），一个 while 循环。

LLM 是一个被关在房间里的天才——他什么都知道，但看不见外面的世界，也碰不到任何东西。bash 工具相当于给他开了一扇窗：他可以对窗外喊一句命令（"帮我看看这个目录有什么文件"），然后从窗外接回结果。while 循环让他可以反复开窗操作，直到任务完成。

run_bash 实现：危险命令检查 + subprocess

def run_bash(command: str) -> str:
    # 黑名单检查：这些命令可能导致系统崩溃
    dangerous = ["rm -rf /", "sudo", "shutdown", "reboot", "> /dev/"]
    if any(d in command for d in dangerous):
        return "Error: Dangerous command blocked"
    try:
        # subprocess.run = Python 调用操作系统执行命令
        r = subprocess.run(command, shell=True, cwd=os.getcwd(),
                           capture_output=True, text=True, timeout=120)
        out = (r.stdout + r.stderr).strip()     # 合并标准输出和错误
        return out[:50000] if out else "(no output)"  # 截断防爆内存
    except subprocess.TimeoutExpired:
        return "Error: Timeout (120s)"             # 超时保护

s02 — Tool Use

"加一个工具, 只加一个 handler" — 循环不用动, 新工具注册进 dispatch map 就行
s02_tool_use.py
149 LOC 4 工具 +42 行

从 1 个工具变成 4 个工具（bash + read_file + write_file + edit_file），但循环代码一行没动。

s01 的实习生只有一把螺丝刀（bash）。s02 给他发了一个工具箱（read、write、edit）。关键是：你不需要重新教他怎么工作（循环不变），只需要把新工具放进工具箱，告诉他 "这个叫 read_file，用来读文件"。

dispatch map：工具名 → 处理函数

# 这就是 Claude Code 所有工具的注册模式：一个字典
TOOL_HANDLERS = {
    "bash":       lambda **kw: run_bash(kw["command"]),
    "read_file":  lambda **kw: run_read(kw["path"], kw.get("limit")),
    "write_file": lambda **kw: run_write(kw["path"], kw["content"]),
    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
}

# 循环中的调用方式（和 s01 完全一样的结构）：
handler = TOOL_HANDLERS.get(block.name)         # 按名字找函数
output = handler(**block.input)                  # 执行

safe_path：路径安全检查

def safe_path(p: str) -> Path:
    path = (WORKDIR / p).resolve()               # 解析为绝对路径
    if not path.is_relative_to(WORKDIR):          # 检查是否在工作目录内
        raise ValueError(f"Path escapes workspace: {p}")
    return path
对比项s01s02
工具数1（bash）4（bash + read + write + edit）
工具调用方式硬编码 run_bash()dispatch map 查找

dispatch map = Claude Code 所有工具的注册模式。无论是内置的 Bash、Read、Write、Edit，还是通过 MCP 协议加载的外部工具，本质上都是往这个字典里加一条记录。这就是为什么循环永远不变。

s04 — Subagent

"大任务拆小, 每个小任务干净的上下文" — 子智能体用独立 messages[], 不污染主对话
s04_subagent.py
184 LOC 5 工具 +task 工具

新增 task 工具 + run_subagent 函数，实现父子代理模式。

主管（父 Agent）把一个调研任务分给下属（子 Agent）。下属拿到一张干净的桌子（独立的 messages=[]），独立完成工作后，只交回一份一句话的总结报告。下属的草稿纸、中间过程全部丢弃。主管的桌面上只多了一行字——保持干净。

run_subagent 核心代码

def run_subagent(prompt: str) -> str:
    sub_messages = [{"role": "user", "content": prompt}]  # 全新的对话历史
    for _ in range(30):          # 安全上限：最多 30 轮
        response = client.messages.create(
            model=MODEL,
            system=SUBAGENT_SYSTEM,  # 子代理有自己的系统提示词
            messages=sub_messages,   # 独立的对话历史
            tools=CHILD_TOOLS,       # 受限的工具集（没有 task 工具）
            max_tokens=8000,
        )
        # ... 标准循环执行工具 ...
    # 只返回最后一条文本，子代理上下文全部丢弃
    return "".join(b.text for b in response.content if hasattr(b, "text"))
    Parent agent                     Subagent
    +------------------+             +------------------+
    | messages=[...]   |             | messages=[]      |  <-- 全新空白
    |                  |  dispatch   |                  |
    | tool: task       | ----------> | while tool_use:  |
    |   prompt="..."   |             |   call tools     |
    |                  |  summary    |   append results |
    |   result = "..." | <--------- | return last text |
    +------------------+             +------------------+
              |
    Parent context stays clean.     子代理上下文被丢弃

上下文隔离 = Agent 架构最重要的设计决策之一

  • 子代理有独立的 messages[]——不会把搜索结果、中间推理塞进父代理的对话
  • 子代理的工具集没有 task——防止无限递归（子代理不能再生子代理）
  • 子代理共享文件系统——可以读写文件，但对话上下文是隔离的
  • 这正是 Claude Code 的 Agent 工具的工作方式

新增三层上下文压缩管线 + .transcripts/ 文件存档。

Agent 的笔记本（messages 列表）会写满。三种应对策略：
Layer 1（擦旧内容）：把旧的工具结果擦掉，只留 "[之前用过 bash]" 这样的标记
Layer 2（拍照存档换新本）：把整本笔记拍照存档到磁盘，让 LLM 写一份摘要，然后换一本新笔记本继续
Layer 3（主动整理）：Agent 自己觉得笔记太乱了，主动调用 compact 工具触发整理

Layer 1：micro_compact 代码 
def micro_compact(messages: list) -> list:
    # 收集所有 tool_result 条目
    tool_results = []
    for msg_idx, msg in enumerate(messages):
        if msg["role"] == "user" and isinstance(msg.get("content"), list):
            for part_idx, part in enumerate(msg["content"]):
                if isinstance(part, dict) and part.get("type") == "tool_result":
                    tool_results.append((msg_idx, part_idx, part))
    # 只保留最近 3 条，旧的替换为占位符
    to_clear = tool_results[:-3]    # KEEP_RECENT = 3
    for _, _, result in to_clear:
        if len(result.get("content", "")) > 100:
            result["content"] = f"[Previous: used {tool_name}]"

def auto_compact(messages: list) -> list:
    # 1. 保存完整对话到磁盘
    TRANSCRIPT_DIR.mkdir(exist_ok=True)
    transcript_path = TRANSCRIPT_DIR / f"transcript_{int(time.time())}.jsonl"

    def _clear_dependency(self, completed_id):

                task["blockedBy"].remove(completed_id)
                self._save(task)

class BackgroundManager:
    def run(self, command: str) -> str:
        task_id = str(uuid.uuid4())[:8]
        # 启动守护线程执行命令
        thread = threading.Thread(
            target=self._execute, args=(task_id, command), daemon=True
        )
        thread.start()
        return f"Background task {task_id} started"  # 立即返回

    def _execute(self, task_id, command):
        # 线程中执行命令，完成后推入通知队列
        r = subprocess.run(command, shell=True, ...)
        with self._lock:
            self._notification_queue.append({...})

    def drain_notifications(self) -> list:
        # 取出并清空所有待处理通知
        with self._lock:
            notifs = list(self._notification_queue)
            self._notification_queue.clear()
        return notifs

# 在 agent_loop 中，每次 LLM 调用前注入通知：
notifs = BG.drain_notifications()
if notifs:
    messages.append({"role": "user",
        "content": f"<background-results>\n{notif_text}\n</background-results>"})

队友做事不能完全无监督。两个协议：

两个协议用的是同一个模式：发请求带 request_id，响应带同一个 request_id 回关联。

request_id 关联模式

# 发起请求时生成唯一 ID
req_id = str(uuid.uuid4())[:8]

# 发送请求到队友邮箱

         "shutdown_request", {"request_id": req_id})

         "shutdown_response", {"request_id": req_id, "approve": True})

# 领导通过 request_id 找到对应的请求，更新状态

request_id 模式是分布式系统的经典模式。当多个参与方异步通信时，你需要一个唯一标识符来关联"请求"和"响应"。HTTP 请求用 Request ID，微服务用 Correlation ID，这里用 uuid。模式相同。

多个队友同时改同一个目录下的文件，一定会冲突。解决方案：给每个队友一间独立办公室（git worktree）。每间办公室是仓库的一个完整副本，在自己的副本上改，改完合并回来。

关键概念拆分：
控制面 = Task（"做什么"——修复登录 bug）
执行面 = Worktree（"在哪做"——/path/to/.worktrees/fix-login/）
两者通过 task_id 绑定。

    def create(self, name, task_id=None, base_ref="HEAD"):
        # 1. 创建 git worktree（一个独立目录 + 独立分支）
        self._run_git(["worktree", "add", "-b", branch, str(path), base_ref])
        # 2. 写入 index.json（记录 name/path/branch/task_id/status）
        idx["worktrees"].append(entry)
        # 3. 如果指定了 task_id，绑定到任务
        if task_id:
            self.tasks.bind_worktree(task_id, name)
        # 4. 发出生命周期事件
        self.events.emit("worktree.create.after", ...)

    def run(self, name, command):
        # 在指定 worktree 的目录中执行命令
        r = subprocess.run(command, shell=True, cwd=path, ...)

    def remove(self, name, force=False, complete_task=False):
        # 1. 删除 git worktree
        self._run_git(["worktree", "remove", path])
        # 2. 如果 complete_task=True，同时把绑定的任务标记为完成
        if complete_task and task_id:
            self.tasks.update(task_id, status="completed")

控制面 vs 执行面 — 这是 s12 最核心的架构拆分：

  • 执行面（Worktree）管环境：代码在哪个目录、哪个分支、怎么运行命令
