01
Prompt Engineering 提示工程
掌握与 Claude Code 高效协作的核心方法论
Claude Code 的核心工作流是 Explore → Plan → Code → Commit，先研究再规划再编码再提交。研究和规划阶段越充分，代码质量越高。

工作流 1：Explore → Plan → Code → Commit

🔍 Explore
阅读文件/图片/URL
用子代理验证
此阶段不写代码
→
📋 Plan
让 Claude 制定计划
使用 think/ultrathink
保存计划以供参考
→
💻 Code
实现解决方案
边写边验证
分步骤推进
→
✅ Commit
提交结果
创建 PR
更新文档
▶ think 思考模式详解

在 Plan 阶段，你可以使用以下关键词增加 Claude 的推理深度：

  • "think" — 基础思考，适合简单问题
  • "think hard" — 深度思考，适合中等复杂度
  • "think harder" — 更深层推理
  • "ultrathink" — 最大计算深度，适合复杂架构决策
推理深度越高，消耗的 token 越多，但对复杂问题的解决质量提升显著。
▶ Plan Mode vs Accept Edits Mode

Claude Code 有两个默认模式，可用 Shift + Tab 切换：

  • Plan Mode（规划模式） — Claude 只分析和规划，不直接修改代码
  • Accept Edits Mode（接受编辑模式） — Claude 可以直接编辑你的文件
对于复杂任务，建议先在 Plan Mode 下规划，确认方案后再切换到 Accept Edits Mode 执行。

02
CLAUDE.md 项目记忆
让 Claude 记住你的项目规范和偏好
CLAUDE.md 是 Claude Code 的项目级记忆文件，每次对话都会自动加载。用 /init 命令创建，持续迭代优化内容是关键。

创建 CLAUDE.md

bash
# 在 Claude Code 中运行
/init

# 这将在项目根目录自动创建 CLAUDE.md

应该写什么内容

  • 项目技术栈 — 使用的语言、框架、工具链
  • 编码规范 — 命名风格、代码格式化工具、lint 规则
  • 项目结构 — 目录含义、关键文件说明
  • 常用命令 — 构建、测试、部署命令
  • 偏好设置 — 你希望 Claude 遵循的具体约定
  • 当前上下文 — 正在开发的功能、已知 bug 等

层级结构

位置作用范围适用场景
项目根目录/CLAUDE.md整个项目项目通用规范
子目录/CLAUDE.md该子目录模块特定规则
~/.claude/CLAUDE.md所有项目个人通用偏好
  • 经常回顾和迭代 CLAUDE.md 内容
  • 保持简洁精炼，像优化 prompt 一样优化它
  • 加入具体示例帮助 Claude 理解
  • 记录测试/构建命令方便自动化
  • 一次性写入大量内容后不再更新
  • 加入无关或过时的信息
  • 把 CLAUDE.md 当成项目文档
  • 忘记提交到版本控制
最常见的错误：写了很多内容但从不迭代优化。CLAUDE.md 应该像频繁使用的 prompt 一样不断打磨，实验什么指令对模型最有效。

# 3. 常用命令（npm run dev / test / build）

自定义斜杠命令

你可以将常用的 Prompt 保存为 Markdown 文件，变成可复用的斜杠命令：

bash
# 创建命令目录
mkdir -p .claude/commands

# 创建自定义命令文件
# 文件名就是命令名：optimize.md → /optimize
echo "分析代码的性能问题并给出优化建议：" > .claude/commands/optimize.md
▶ 自定义命令示例：pull-request.md
markdown
# Create Pull Request Command

Create a new branch, commit changes, and submit a pull request.

## Behavior
- Creates a new branch based on current changes
- Formats modified files using Biome
- Analyzes changes and automatically splits into logical commits
- Each commit focuses on a single logical change or feature
- Creates descriptive commit messages for each logical unit
- Pushes branch to remote
- Creates pull request with proper summary and test plan

## Guidelines for Automatic Commit Splitting
- Split commits by feature, component, or concern
- Keep related file changes together in the same commit
- Separate refactoring from feature additions
- Ensure each commit can be understood independently
输入 / 可看到所有可用命令的自动补全列表，包括内置和自定义命令。

Claude Skills 技能系统
将复杂工作流封装为可复用的斜杠命令
Skills（技能）= 自定义斜杠命令。将 Markdown 文件放入 .claude/commands/ 目录，即可用 /技能名 随时调用。它们是"AI 的宏"——定义一次，反复使用。

快速上手（3 分钟）

bash
# 1. 创建技能目录
mkdir -p .claude/commands

# 2. 创建一个简单的技能文件
cat > .claude/commands/analyze.md << 'EOF'
# Code Analysis

Analyze the current code for:
- Potential bugs and edge cases
- Performance optimizations
- Code quality improvements
- Security vulnerabilities

Provide specific, actionable recommendations.
EOF

# 3. 在 Claude Code 中使用
# 输入: /analyze

7 个预置技能详解

技能类型功能描述
/pr工作流自动创建分支、拆分 commit、推送并创建 Pull Request
/tdd工作流严格的 Red-Green-Refactor 测试驱动开发周期
/review质量六角色代码审查（PM、Dev、QA、安全、DevOps、UX）
/test质量单元测试最佳实践清单
/five方法论五个为什么根因分析法

05
Memory 记忆系统
两种记忆方案：内置 /memory 命令 vs MCP Memory Server
Claude Code 有两套记忆方案：内置的 /memory 命令编辑 CLAUDE.md 文件，以及 MCP Memory Server 提供跨会话持久记忆。两者可以互补使用。

方案对比

特性/memory 命令MCP Memory Server
原理编辑 CLAUDE.md 文件独立的 MCP 服务器存储
持久化文件系统（随项目）跨会话持久存储
范围项目级全局或项目级可选
安装内置，无需安装需要安装 MCP 服务器
用途项目规范、编码约定用户偏好、跨项目记忆
版本控制可提交到 Git独立存储，不进入 Git

安装 MCP Memory Server

bash
# 全局安装（推荐：跨项目持久记忆）
claude mcp add memory -s user -- npx -y @modelcontextprotocol/server-memory

# 项目级安装（项目隔离记忆）
claude mcp add memory -s local -- npx -y @modelcontextprotocol/server-memory

# 验证安装
claude mcp list
# 期望输出: memory: npx -y @modelcontextprotocol/server-memory - ✓ Connected

使用场景

  • 记住项目编码规范 — "记住当前项目使用 TypeScript"
  • 存储常用配置 — "记住我偏好使用 pnpm 而不是 npm"
  • 维护任务上下文 — "记住我们正在重构认证模块"
  • 跟踪决策记录 — "记住上次会议决定使用 PostgreSQL"
全局安装允许所有项目共享记忆，项目级安装则将记忆隔离在单个项目中。建议：全局装一个用于通用偏好，重要项目再装项目级的。

07
多窗口 Git Worktree
同时在多个分支上运行独立的 Claude Code 会话
Git Worktree 允许同一个 Git 仓库在不同文件夹中打开不同分支，每个文件夹可以运行独立的 Claude Code 会话，互不干扰。

核心概念

  • 正常 Git — 一个仓库同时只能在一个文件夹、一个分支
  • Worktree — 可以同时在多个文件夹、多个分支工作
  • 隔离性 — 一个 worktree 的改动不影响其他 worktree

操作步骤

bash
# 1. 创建 Worktree（在项目根目录执行）
git worktree add -b feature-a ../feature-a
git worktree add -b feature-b ../feature-b

# 2. 在每个 Worktree 中启动 Claude
cd ../feature-a && claude    # 终端 Tab 1
cd ../feature-b && claude    # 终端 Tab 2

# 3. 每个终端中的 Claude 独立工作在各自的分支上

# 4. 完成后清理
git worktree remove ../feature-a
git worktree remove ../feature-b
主仓库
branch: main
Claude 会话 1
→
Worktree A
branch: feature-a
Claude 会话 2
→
Worktree B
branch: feature-b
Claude 会话 3
▶ 使用 Tmux 管理多个会话

使用 Tmux 可以创建持久化的终端会话，即使断开连接也能继续运行：

bash
# 创建命名会话
tmux new-session -s feature-a
cd ../feature-a && claude

# 分离会话（Ctrl+B, D）
# 重新连接
tmux attach -t feature-a
  • 使用一致的命名规范
  • 每个 worktree 一个终端标签
  • 用 Tmux 管理后台会话
  • 用完及时清理 worktree
  • 创建过多 worktree 导致混乱
  • 忘记清理已完成的 worktree
  • 在同一 worktree 开多个 Claude
  • 忽略 token 共享消耗（多会话共享额度）
所有 Claude Code 会话共享同一个 token 配额。多个终端并行运行会更快消耗你的额度——3 个并行会话大约每个会话只有 3-13 个 prompt。

09
Agent Teams 团队协作
实验性多代理协作：Team Lead + Teammates 结构
Agent Teams 是实验性功能，在单个 Claude Code 会话中实现多代理协作。Team Lead 负责协调，Teammates 专注各自任务，通过共享任务列表同步。

启用方法

bash
# 启用 Agent Teams（需要环境变量）
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 永久配置（添加到 shell 配置文件）
echo 'export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1' >> ~/.zshrc
source ~/.zshrc

# 验证
echo $CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS  # 应输出: 1

# 启动 Claude Code
claude

团队结构

  • Team Lead（主代理）— 创建/分配任务、监控进度、做架构决策、协调 Teammates、审查成果
  • Teammates（专业代理）— 专注分配的任务、报告进度和阻塞、可并行工作、更新共享任务列表
  • Shared Task List（共享任务列表）— 中心协调点、全员可见、追踪依赖关系、防止重复工作

4 种协作模式

▶ 1. 并行开发

Task 2: 后端工程师 - 实现认证 API

三个代理同时工作。

  ↓ (完成后)
Task 2: 安全审查者 - 检查漏洞
Task 3: 性能审查者 - 分析优化点

▶ 4. 研究 + 实施

  ↓ (生成建议)

  - Teammate A: 更新认证模块
  - Teammate B: 更新支付模块
  - Teammate C: 更新通知模块
Prompt
我需要重构认证系统。使用 agent teams：

1. 创建 "research" teammate 分析现有认证实现
2. 创建 "backend" teammate 重构认证逻辑
3. 创建 "frontend" teammate 更新 UI 组件
4. 创建 "qa" teammate 编写全面测试

协调它们按顺序工作，每个将发现传递给下一个。
  • 分配清晰、聚焦的任务
  • 使用描述性名称: "frontend-specialist"

Token 与用量管理
理解 Token 计算、省量技巧、计划对比
Token 是 AI 处理文本的基本单位。你的每一条 prompt、Claude 的回复、读取的文件、加载的上下文都消耗 Token。1000 tokens ≈ 750 个英文单词。

Token 计算方式

  • 文本公式：tokens ≈ 单词数 + 标点符号数
  • 图片公式：tokens = (宽 px × 高 px) / 750
  • 粗略换算：1 token ≈ 0.75 个英文单词
  • 中文：1 个汉字通常 = 1-2 个 token

什么消耗 Token

  • 你的 prompt 和问题
  • Claude 的回复和解释
  • 系统指令和工具定义
  • Claude 读取的文件内容
  • 项目结构分析
  • 图片（按像素面积计算）

新手常见错误汇总
从全文提取的所有 Don't / Warning / Common Issues
以下是从完整教程中提取的所有常见错误和陷阱。避开这些坑，你的 Claude Code 体验会顺畅 10 倍。
1. CLAUDE.md 不迭代
写了大量内容但从不回顾优化。CLAUDE.md 应像 prompt 一样持续打磨。
2. 复杂任务不先规划
直接让 Claude 写代码而不走 Explore → Plan → Code 流程，导致质量下降。
3. 运行不受信任的 Skills
Skills 有完整代码库权限，可执行任意命令。克隆仓库后必须先审查 .claude/commands/ 目录。
4. 多会话不注意额度
所有 Claude Code 会话共享同一 token 配额。3 个并行会话会迅速耗尽额度。
5. 简单任务用 Fast Mode
Fast Mode 价格是标准的 6 倍。用于后台任务、文档等非紧急场景纯属浪费。
6. Agent Teams 团队过大
创建太多 Teammates 导致协调开销超过收益。最优规模 3-5 个。
7. 跳过任务依赖
在 Agent Teams 中不设置 blockedBy 关系，导致代理间冲突和重复工作。
8. Hooks 脚本不审查
Hooks 执行任意 shell 命令，不验证脚本就配置可能导致安全问题。
9. 技能命名不规范
使用 command.md 这样模糊的名称。应使用描述性名称如 create-api-endpoint.md。
10. 不使用 /compact 压缩
长对话不压缩导致上下文膨胀，浪费 token 且降低回复质量。
11. 一个技能做太多事
技能应单一职责。一个做 10 件事的技能本质上是 10 个技能的糟糕合体。
12. 忘记 Worktree 清理
用完 Git Worktree 后不清理，留下大量无用目录。用 git worktree remove 及时清理。
13. 全局 Skills 权限过大
放在 ~/.claude/commands/ 的技能会在所有项目中执行。只添加完全信任的技能。
14. 混淆 /login 和 /auth login
2026 年起应使用新的 /auth login 命令代替旧的 /login。
