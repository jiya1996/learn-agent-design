3. 模块 1：Agent Loop — 核心中的核心

文件路径：internal/llm/agent/agent.go (759 行)

核心模块
想象你和一个很聪明的助手对话。你说"帮我查一下 X 文件里有没有 bug"，助手想了想说"我需要先看看这个文件"（工具调用请求），你把文件内容给他看（工具执行结果），他又说"我发现第 42 行有问题，我需要修改它"（又一个工具调用），你帮他改了（执行），他说"好了，改完了，这是修改说明"（最终回复）。

这个"对话 → 思考 → 需要工具 → 执行工具 → 把结果告诉 AI → 再思考 → ..." 的循环就是 Agent Loop。

核心循环伪代码

函数 processGeneration(用户消息):

    消息历史 = 获取之前的所有对话记录
    消息历史.追加(用户消息)

    // 如果是第一条消息，后台生成会话标题
    如果 消息历史为空:
        后台启动: 生成标题(用户消息)

    // ★ 这就是 Agent Loop 的核心
    无限循环:

        // 第一步：把消息历史发给 LLM
        AI回复, 工具调用列表 = 调用LLM(消息历史)

        // 第二步：检查 AI 是否需要使用工具
        如果 AI回复说"我要用工具" 且 工具调用列表不为空:

            // 第三步：逐个执行工具
            对于每个工具调用:
                找到对应的工具对象
                工具结果 = 工具.执行(参数)

            // 第四步：把工具结果加入消息历史，继续循环
            消息历史.追加(AI回复)
            消息历史.追加(工具结果)
            继续循环  ← 回到第一步，让 AI 看到结果后继续思考

        否则:
            // AI 不需要工具了，说明任务完成
            返回 AI回复  ← 循环结束
Agent Loop 的本质就是一个 for 循环。每次迭代做三件事：
1. 把对话历史发给 LLM
2. 如果 LLM 要求使用工具 → 执行工具 → 把结果加入历史 → 继续循环
3. 如果 LLM 不要求工具 → 返回最终回复 → 循环结束

流式处理 (Streaming)

普通模式像发短信——等对方写完整条消息才收到。流式模式像打电话——对方说一个字你就听到一个字，感觉更快、更自然。

在 streamAndHandleEvents 方法中，LLM 的回复不是一次性返回的，而是通过一个 channel（Go 语言的"管道"）一个事件一个事件地流过来：

事件类型含义TUI 的反应
EventThinkingDeltaAI 的思考过程片段显示"思考中..."
EventContentDeltaAI 回复的文字片段逐字显示回复
EventToolUseStartAI 开始请求一个工具显示工具名称
EventToolUseStop工具参数接收完毕准备执行工具
EventComplete整个回复结束统计 token 和费用
EventError出错了显示错误信息

取消机制

每个会话的请求都有独立的"取消开关"（CancelFunc），存在一个线程安全的 Map 里。用户按 Esc 时，调用对应的 CancelFunc，Agent Loop 在下次循环检查时发现已取消，就停止工作。

4. 模块 2：工具系统 — Agent 的"双手"

文件路径：internal/llm/tools/

核心模块
Agent 本身只会"思考"和"说话"，不能直接操作文件系统或执行命令。工具就像它的"双手"——想查文件就用 View 工具，想改文件就用 Edit 工具，想执行命令就用 Bash 工具。每个工具都有说明书（Info），告诉 AI "我能做什么、需要什么参数"。

工具接口定义

// 每个工具必须实现这两个方法：

接口 BaseTool:
    Info()  → ToolInfo    // "我是谁，我能做什么，我需要什么参数"
    Run()   → ToolResponse // "执行！返回结果"

结构 ToolInfo:
    Name        string          // 工具名，如 "bash", "edit"
    Description string          // 给 AI 看的详细说明
    Parameters  map[string]any  // JSON Schema 格式的参数定义
    Required    []string        // 必填参数列表

完整工具清单

工具名文件功能需要权限关键细节
bashbash.go执行 Shell 命令部分需要安全命令（git status、ls 等）免权限；禁止 curl/wget 等网络命令；输出超 30KB 自动截断
editedit.go编辑文件（查找替换）是必须先 view 才能 edit；old_string 必须唯一匹配；自动记录版本历史
writewrite.go写入/创建文件是覆盖整个文件；自动创建父目录；检查文件是否被外部修改
patchpatch.go批量修改多个文件是（每个文件）统一 diff 格式；支持添加/更新/删除文件；每个文件变更单独请求权限
viewview.go查看文件内容否最大 250KB；默认显示前 2000 行；支持 offset/limit 分页；带行号
grepgrep.go搜索文件内容否支持正则表达式；优先用 ripgrep，退回纯 Go；最多返回 100 个结果
globglob.go按文件名模式搜索否支持 ** 通配符；优先用 ripgrep；最多 100 个结果
lsls.go列出目录结构否树形显示；自动跳过 node_modules/.git 等；最多 1000 个文件
fetchfetch.go抓取网页内容是支持 text/markdown/html 三种输出格式；最大 5MB；超时 30 秒

diagnosticsdiagnostics.go获取代码诊断否依赖 LSP 客户端；显示错误/警告/提示；按严重程度排序
agentagent-tool.go启动子代理否子代理只有只读工具；创建独立会话；费用合并到父会话

工具注册机制

文件路径：internal/llm/agent/tools.go

// 主 Agent（CoderAgent）的工具清单 — 有完整权限
CoderAgentTools = [
    Bash, Edit, Write, Patch, Fetch,      // 需要权限的工具
    View, Glob, Grep, Ls, Sourcegraph,    // 只读工具
    Diagnostics, Agent,                    // 辅助工具
    ...MCP工具                             // 外部工具（动态发现）
]

// 子 Agent（TaskAgent）的工具清单 — 只有只读权限
TaskAgentTools = [
    Glob, Grep, Ls, Sourcegraph, View     // 只能看，不能改
]
工具注册的设计要点：
1. 主 Agent 全能，子 Agent 只读 — 这就是"权限隔离"
2. 工具通过 Info() 方法把自己的"说明书"（JSON Schema）告诉 LLM，LLM 根据说明书决定何时调用哪个工具
3. MCP 工具是动态发现的，启动时扫描配置中的 MCP 服务器

文件修改安全

// 防止覆盖外部修改的三步检查：

5. 模块 3：Provider 抽象层

文件路径：internal/llm/provider/ (总计 2559 行)

重要模块
想象你有一个万能充电器，不管是 iPhone、Android、还是笔记本电脑，都能用同一个接口充电。Provider 层就是这个"万能充电器"——不管底层是 Claude、GPT 还是 Gemini，Agent Loop 用同一个接口调用，不需要关心具体是哪个模型。

接口定义

接口 Provider:
    SendMessages(消息列表, 工具列表) → 回复      // 一次性发送
    StreamResponse(消息列表, 工具列表) → 事件流   // 流式返回
    Model() → 模型信息                           // 当前模型

工厂模式：一个入口，多种实现

函数 NewProvider(提供商名称, 选项...):
    根据 提供商名称:
        "anthropic"  → 创建 AnthropicClient   // Claude API
        "openai"     → 创建 OpenAIClient      // OpenAI API
        "gemini"     → 创建 GeminiClient       // Google API
        "copilot"    → 创建 CopilotClient      // GitHub Copilot
        "bedrock"    → 创建 AnthropicClient    // AWS 版 Claude（代理模式）
        "azure"      → 创建 OpenAIClient       // Azure 版 OpenAI（包装模式）
        "vertexai"   → 创建 GeminiClient       // Google Cloud 版 Gemini
        "groq"       → 创建 OpenAIClient       // 换 URL 即可（兼容 OpenAI）
        "openrouter" → 创建 OpenAIClient       // 换 URL 即可
        "xai"        → 创建 OpenAIClient       // 换 URL 即可
        "local"      → 创建 OpenAIClient       // 本地模型
巧妙的复用设计：
- GROQ、OpenRouter、XAI、Local 都兼容 OpenAI 的 API 格式，所以直接复用 OpenAIClient，只改 URL
- Azure 包装了 OpenAI 客户端，加上 Azure 认证
- VertexAI 复用了 Gemini 客户端，改了后端地址
- Bedrock 是代理模式，内部委托给 Anthropic 客户端

每个 Provider 的核心差异

Provider消息格式工具格式特殊功能
AnthropicContent Block 数组ToolUseBlockExtended Thinking（深度思考）、Prompt Caching（缓存节省费用）
OpenAIMessage 对象Function 定义Reasoning Effort（推理力度调节）
GeminiContent + PartsFunctionDeclaration需要递归类型转换（最复杂的 Schema 处理）
Copilot同 OpenAI同 OpenAIGitHub Token 自动刷新、多工具流式适配

统一的重试机制

所有 Provider 共享的重试策略：
    最大重试次数 = 8
    退避公式 = 2000毫秒 × 2^(第几次重试)   // 2秒, 4秒, 8秒, 16秒...
    随机抖动 = ±20%                        // 避免所有请求同时重试

    重试条件：
    - 429 = 速率限制（太多请求）
    - 500 = 服务器错误
    - 529 = 服务过载（Anthropic 特有）
    - 401 = 认证失效（Copilot 会自动刷新 token）

6. 模块 4：权限模型 — 安全门卫

文件路径：internal/permission/permission.go (119 行)

重要模块
像酒店的房卡系统：工具要进入"危险区域"（写文件、执行命令）时，必须先到前台（Permission Service）申请通行证。前台广播给住客（用户）"有人要进 312 号房，允许吗？"，住客可以说"允许"、"这次都允许"或"拒绝"。
工具想执行 → permission.Request() │ ├─ 检查 1：是否自动批准模式？（非交互模式） │ └─ 是 → 直接放行 │ ├─ 检查 2：是否有相同操作的历史批准？（同工具+同操作+同路径+同会话） │ └─ 是 → 直接放行 │ └─ 都没有 → 发布事件到 PubSub → TUI 弹出权限对话框 │ ├─ 用户点 "Allow(a)" → 本次批准 ├─ 用户点 "AllowSession(s)" → 同类操作都批准 └─ 用户点 "Deny(d)" → 拒绝

权限请求的数据结构

一个权限请求包含：
    ID          = 唯一标识
    SessionID   = 哪个会话
    ToolName    = 哪个工具（bash/edit/write/fetch/patch）
    Action      = 什么操作（execute/write/fetch）
    Description = 人类可读的描述（如 "Execute: grep -r 'bug' src/"）
    Path        = 操作的路径
    Params      = 详细参数（包括 diff 预览）

7. 模块 5：Session 与 Message — 对话记忆

文件路径：internal/session/session.go + internal/message/message.go

Session 像一个"对话文件夹"，Message 像文件夹里的"每一页纸"。你可以有多个文件夹（多个会话），每个文件夹里按时间顺序存放对话记录。

Session（会话）

一个会话包含：
    ID               = 唯一标识
    ParentSessionID  = 父会话（子代理会话的父级）
    Title            = 标题（自动生成）
    MessageCount     = 消息数量
    PromptTokens     = 输入 token 数
    CompletionTokens = 输出 token 数
    Cost             = 费用（美元）
    SummaryMessageID = 压缩摘要的消息 ID

Message（消息）

消息有三种角色：

角色谁发的典型内容
user用户文字 + 可能的图片附件
assistantAI文字回复 + 思考过程 + 工具调用请求
tool工具系统工具执行的结果

每条消息的内容是一个 Parts 数组，可以包含多种类型：

一条消息的 Parts 示例：
[
  {类型: "text",       数据: {text: "请帮我查看 main.go"}},
  {类型: "tool_call",  数据: {id: "1", name: "view", input: '{"file_path": "main.go"}'}},
  {类型: "tool_result", 数据: {tool_call_id: "1", content: "package main..."}},
  {类型: "finish",     数据: {reason: "end_turn", time: 1234567890}}
]

上下文压缩（Summarize）

对话太长了怎么办？就像开会记录太多页了，安排一个秘书（Summarizer Agent）把前面的内容浓缩成一段摘要，后续对话只看摘要 + 最新消息，省下大量 token。 
压缩流程：
1. 对话达到上下文窗口的 95% 时自动触发
2. 把所有消息发给 Summarizer Agent，让它生成摘要
3. 保存摘要消息，记录 SummaryMessageID
4. 后续对话从摘要消息开始，跳过更早的消息

8. 模块 6：系统提示词 — Agent 的"性格说明书"

9. 模块 7：子代理 — Agent 中的 Agent

文件路径：internal/llm/agent/agent-tool.go (109 行)

主 Agent 就像项目经理，当遇到"搜索代码库里所有用到 config 的地方"这种查询任务时，与其自己一个个文件找，不如派一个实习生（子代理）去做——实习生只能看不能改，找完了把报告交给经理，经理再决定下一步。
主 Agent 调用 agent 工具 │ ├─ 创建新的 TaskAgent（只有只读工具） ├─ 创建子 Session（关联到父 Session） ├─ 执行 agent.Run(子Session, 查询指令) │ │ │ └─ 子 Agent Loop（独立循环） │ ├─ 可以用: Glob, Grep, Ls, View, Sourcegraph │ └─ 不能用: Bash, Edit, Write, Patch, Fetch │ ├─ 等待子 Agent 完成 ├─ 获取子 Agent 的回复 ├─ 合并费用到父 Session └─ 把回复作为工具结果返回给主 Agent
子代理的三个设计原则：
1. 只读隔离：子代理不能修改任何文件，只能查询
2. 上下文隔离：子代理有自己独立的会话，不污染主对话
3. 费用透明：子代理的费用自动合并到父会话的统计中

10. 模块 8：MCP 协议 — 外部工具接入

文件路径：internal/llm/agent/mcp-tools.go (202 行)

知识模块
MCP（Model Context Protocol）就像 USB 接口——你可以插入各种外部设备（MCP 服务器），每个设备提供不同的工具（搜索 Jira、查数据库、操作 Playwright 等）。Open Code 会自动发现这些设备上有哪些工具可用，然后把它们无缝地加入到 Agent 的工具箱里。

MCP 工具发现流程

1. 读取配置中的 MCP 服务器列表

2. 对每个 MCP 服务器：
   ├─ 如果是 Stdio 类型 → 启动子进程，通过标准输入/输出通信
   └─ 如果是 SSE 类型  → 连接 HTTP 服务器，通过事件流通信

3. 发送握手请求：
   "我是 OpenCode，版本 X，支持 MCP 协议版本 Y"

4. 发送 ListTools 请求：
   "你有哪些工具？"
   → 服务器返回工具列表，每个包含名称、描述、参数 Schema

5. 包装每个 MCP 工具为 OpenCode 的 BaseTool：
   MCP 工具名 "write_file" → OpenCode 名 "anthropic_mcp_write_file"

6. 添加到主 Agent 的工具清单中

MCP 工具执行流程

AI 调用 MCP 工具 →
    请求用户权限 →
    创建 MCP 客户端 →
    发送 CallTool 请求 →
    接收结果 →
    关闭客户端 →
    返回结果给 Agent

11. 模块 9：TUI 界面 — 终端里的"App"

文件路径：internal/tui/tui.go (966 行)

知识模块
TUI 用的是 Bubble Tea 框架——Go 语言最流行的终端 UI 框架。它的设计理念和 React 很像：数据变了 → 自动重新渲染。但不同于网页，它渲染的是终端里的文字和颜色。

Bubble Tea 的三段式生命周期

Init()     → 初始化：创建所有子组件（聊天页、对话框、状态栏...）

Update(msg) → 响应事件：
    收到键盘事件？  → 分发给对应组件
    收到窗口大小变化？ → 通知所有组件调整布局
    收到新消息事件？ → 更新消息列表、滚动到底部
    收到权限请求？  → 弹出权限对话框

View()     → 渲染：
    1. 渲染底层页面（聊天或日志）
    2. 加上状态栏
    3. 如果有对话框打开 → 叠加在上面（居中显示）

12. 模块 10：事件总线 — 模块间的"广播系统"

文件路径：internal/pubsub/broker.go

事件总线就像广播电台：Agent 在后台处理时，产生各种事件（新消息、权限请求、完成通知），广播出去。TUI 的各个组件各自"收听"自己关心的频道，收到广播后更新界面。发送方和接收方互不认识，完全解耦。 
事件类型只有三种：
    Created = "创建了"  // 新消息、新会话、新权限请求
    Updated = "更新了"  // 消息内容更新（流式输出时频繁触发）
    Deleted = "删除了"  // 会话删除

配置：
    频道缓冲区 = 64 条事件
    最大事件数 = 1000
    发布模式 = 非阻塞（缓冲区满了就丢弃）

订阅者收到的数据：
    Event[T] = {
        Type: "created" | "updated" | "deleted",
        Payload: T  // 具体数据（消息、会话、权限请求等）
    }

事件流转全景

Agent Loop 执行 │ ├─ 创建消息 → message.Service.Create() │ └─ Broker.Publish(Created, 新消息) │ └─ TUI 消息列表收到 → 添加并渲染 │ ├─ 流式更新 → message.Service.Update() │ └─ Broker.Publish(Updated, 更新的消息) │ └─ TUI 消息列表收到 → 重新渲染该消息 │ ├─ 需要权限 → permission.Service.Request() │ └─ Broker.Publish(Created, 权限请求) │ └─ TUI 弹出权限对话框 │ └─ Agent 完成 → agent.Publish(Created, AgentEvent) └─ TUI 状态栏更新、检查是否需要压缩
