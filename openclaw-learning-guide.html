02
Agent 循环
run.ts 的重试循环逐行解析

"不断重试，直到成功或彻底放弃 — 这就是 Agent 的生存哲学"

想象一个客服打电话给专家（LLM）请教问题。如果专家那边占线（overload），客服会等一会儿再拨；如果专家的手机没信号（auth 过期），客服换一个号码拨；如果双方说到一半对话记录太多本子写不下了（context overflow），客服把前面的内容总结一下，从摘要继续。这就是 run.ts 的核心逻辑。

重试循环的宏观流程

runEmbeddedPiAgent()
    │
    ├─ 初始化：workspace、session、auth profiles
    │
    ├─ 计算最大重试次数：
    │   BASE = 24 + profiles × 8 = 最少 32，最多 160
    │
    └─ while (attempt < maxRetries):
        │
        ├─→ runEmbeddedAttempt()   ← 单次 LLM API 调用
        │       │
        │       ├─ 构建 system prompt
        │       ├─ 准备 tools 清单
        │       ├─ 调用 LLM 流式 API
        │       └─ 返回结果 or 错误
        │
        ├─ 检查结果 ─┬─ 成功 → 提取 tool calls → 执行 → 累计 usage → 继续
        │            │
        │            ├─ billing error → 记录，退出
        │            │
        │            ├─ auth error → 刷新 token / 切换 profile → 重试
        │            │
        │            ├─ overload → 指数退避 (250ms → 1.5s) → 重试
        │            │
        │            ├─ timeout → 重试
        │            │
        │            └─ context overflow → 触发 compact.ts → 压缩后重试
        │
        └─ 累计 token usage（input + output + cache read/write）

关键代码注释

重试常量

TypeScript
// 基础重试次数 — 无论有多少 auth profile，至少重试 24+8=32 次
const BASE_RUN_RETRY_ITERATIONS = 24;
const RUN_RETRY_ITERATIONS_PER_PROFILE = 8;

// 退避策略：服务端 overload 时，每次等待时间递增
const OVERLOAD_FAILOVER_BACKOFF_POLICY = [
  250,   // 第 1 次：等 0.25 秒
  500,   // 第 2 次：等 0.5 秒
  750,   // 第 3 次：等 0.75 秒
  1000,  // 第 4 次：等 1 秒
  1500,  // 第 5 次及以后：等 1.5 秒
];

Open Code 的 Agent 循环

processGeneration 的 for 循环

简洁：调 LLM → 有工具就执行 → 没工具就结束

无 auth 轮转（单用户）

无退避策略

OpenClaw 的 Agent 循环

runEmbeddedPiAgent 的 while 循环

复杂：32-160 次重试 + 多 profile 轮转

指数退避 250ms → 1.5s

1700 行（含所有错误处理）

核心洞察：Open Code 的循环是"理想路径"（happy path），OpenClaw 的循环是"生产路径"（production path）。 差别不在于 LLM 调用本身，而在于所有可能出错的地方都有对策：auth 过期、费用超限、服务过载、上下文溢出。 这就是"玩具项目"和"生产系统"的本质区别。
▶深入：单次尝试 attempt.ts 做了什么

runEmbeddedAttempt() 是循环体内每次调用的核心函数：

  1. 准备 session — 组装对话历史（messages）、可用工具列表、system prompt
  2. 调 LLM 流式 API — 发送请求，逐 token 接收回复
  3. 解析响应 — 从流中提取 tool_use 调用和文本回复
  4. 处理 provider 差异 — Anthropic、OpenAI、Google 的 API 格式不同，这里做统一适配
  5. 记录 token 用量 — 返回本次调用消耗的 token 数

工具系统
注册、Profile 分级、7 层策略管道

"Agent 的能力边界 = 它能使用的工具集合"

想象一个酒店的客房服务菜单。不同等级的房间看到的菜单不同：普通房（minimal）只有基本餐点，商务房（coding）加了咖啡和打印服务，总统套房（full）什么都能点。而且，酒店安保部门还有一份"禁止清单"— 某些物品不论什么房间都不让送。这就是 OpenClaw 的工具策略管道。

04
多频道路由
消息从 Telegram 到 Agent 的完整旅程

"一个 Agent，同时活在 20 个世界里"

想象一家全球连锁酒店。旅客从不同入口（机场接驳、出租车、地铁）到达前台。前台根据预约信息（binding）找到对应的客房管家（Agent）。管家记得每位旅客的偏好（session），正在为一位客人服务时会挂出"忙碌中"的牌子（state machine）。这就是 OpenClaw 的多频道路由系统。

一条消息的完整旅程

用户在 Telegram 发送 "帮我看看这段代码"
    │
    ▼
[Channel Plugin: Telegram]
    │  解析 message、sender、chat_id
    ▼
[resolve-route.ts] → 查 binding 表：
    │  1. binding.peer      → 有没有专门绑定给这个用户？
    │  2. binding.guild     → Discord 服务器级别绑定？
    │  3. binding.team      → Teams 团队级别？
    │  4. binding.account   → 账号级别？
    │  5. binding.channel   → 频道级别？
    │  6. default           → 兜底：用默认 Agent
    │
    ▼
[ResolvedAgentRoute]
    │  agentId: "claw"
    │  sessionKey: "agent:claw:channel:telegram:peer:user:12345"
    │  matchedBy: "binding.peer"
    │
    ▼
[session.ts] → recordInboundSession()
    │  记录会话、更新最后路由信息
    │
    ▼
[run-state-machine.ts]
    │  onRunStart() → activeRuns++, 开始心跳
    │
    ▼
[runEmbeddedPiAgent()] → Agent 循环处理 → 生成回复
    │
    ▼
[run-state-machine.ts]
    │  onRunEnd() → activeRuns--, 停止心跳
    │
    ▼
[Channel Plugin: Telegram] → 把回复发回 Telegram

Session Key 格式

TypeScript
// Session Key = Agent 和用户之间的唯一对话标识
// 格式：agent:{agentId}:channel:{channel}:peer:{peerKind}:{peerId}

// 例子：
"agent:claw:channel:slack:peer:user:U123"     // Slack 上的 U123 用户
"agent:claw:channel:telegram:peer:user:12345" // Telegram 上的 12345 用户
"agent:claw:channel:discord:peer:guild:G789"  // Discord 服务器级别

// DM 作用域选项 — 控制会话隔离粒度：
// "main"                → 所有人共享一个会话（像公共聊天室）
// "per-peer"            → 每人一个会话（像私人助理）
// "per-channel-peer"    → 每个频道×每人 一个会话
// "per-account-channel-peer" → 最细粒度

    │
    ▼

核心洞察：Open Code 和 Learn Claude Code 都只有一个"频道"— 终端。 OpenClaw 的多频道架构是它最独特的设计：同一个 Agent 同时服务于 20+ 平台。 这引入了 Open Code 完全不需要考虑的问题：路由决策、会话隔离、状态追踪、心跳保活。
▶深入：Channel Plugin 的插件架构

每个频道是一个独立插件，拥有：

  • 消息接入管道 — 接收并解析该平台格式的消息
  • 回复处理器 — 把 Agent 输出转换成该平台格式发回
  • 账号/凭证管理 — 存储 bot token、OAuth 密钥等
  • 频道特有能力声明 — 比如 Discord 支持 reaction 和 thread，但 IRC 不支持

插件通过 channels/plugins/bundled.ts 注册，运行时懒加载——只初始化用户配置了的频道，避免浪费资源。

上下文与记忆
短期压缩 + 长期向量数据库

"短期记忆让 Agent 能对话，长期记忆让 Agent 能成长"

想象一个医生。短期记忆是今天看诊时的病历笔记——写满一页就总结上半页、翻到新页继续（context compaction）。长期记忆是医院的电子病历系统——通过关键词搜索以前的就诊记录（vector embedding search）。OpenClaw 两个都有，而 Open Code 只有短期的。

Skills 与扩展
52 个 Skill 分类、Extension 机制

"Skills 不是工具，是知识 — 教 Agent 如何使用工具的说明书"

想象一个新员工入职。公司不可能把所有部门的操作手册都塞给他（context 会爆），而是给他一本"目录手册"，列出所有可用的操作指南。当他需要某个特定操作时，去书架上找对应的手册借来看。Skill 就是这些操作手册——目录放在 system prompt 里，全文通过 tool_result 按需加载。

Skill 加载机制

Agent 初始化时：
    │
    ├─ 扫描 skills/ 目录
    ├─ 读取每个 Skill 的 YAML frontmatter（name, description, requires）
    ├─ 检查依赖是否满足（requires.bins 中的命令是否存在）
    └─ 把 Skill 目录（name + description）写入 system prompt
           "你可以使用以下 Skills: github, coding-agent, ..."

Agent 运行时（按需加载）：
    │
    ├─ LLM 判断需要某个 Skill 的知识
    ├─ 通过 tool_result 返回 Skill 全文
    └─ Agent 获得该领域的详细操作指南

安全模型
DM pairing、危险工具清单

"信任，但要验证 — 尤其是在多租户环境中"

想象一座写字楼的安保系统。普通员工（channel user）刷卡只能到自己的楼层；经理（admin）可以到更多区域；只有大楼业主（owner）有万能钥匙。某些区域（如服务器机房）任何人进入都需要额外审批（dangerous tools）。通过 HTTP 远程访问的人比在场的人权限更低（gateway deny）。

危险工具分级

Gateway HTTP 自动禁止的工具

TypeScript
// 通过 HTTP API 远程访问时，这些工具直接禁止
// 原因：RCE 风险、需要终端交互、或在远程场景下无意义
[
  "sessions_spawn",    // 不能远程创建新会话
  "sessions_send",     // 不能远程发送到其他会话
  "cron",              // 不能远程创建定时任务
  "gateway",           // 不能远程控制网关
  "whatsapp_login",    // 需要扫 QR 码，远程做不了
]

ACP（Automation Control Plane）危险工具

TypeScript
// 在自动化场景中（无人值守），这些工具始终需要明确审批
[
  "exec",             // 执行任意命令 — 最高风险
  "spawn",            // 启动新进程
  "shell",            // 打开 shell
  "sessions_spawn",   // 创建新会话
  "sessions_send",    // 向会话发消息

  "fs_write",         // 写文件
  "fs_delete",        // 删文件

  "apply_patch",      // 批量修改代码
]

    │

Open Code 的权限模型

运行时交互式：工具要执行 → 弹窗问用户

3 种批准方式：Grant / GrantPersistent / AutoApprove

单用户，不需要考虑多租户

无危险工具分级

信任边界：用户的终端

OpenClaw 的安全模型

编译时声明式：策略管道预过滤

7 层策略 + 危险工具分级 + Owner-Only

多租户，不同用户看到不同工具

Gateway / ACP 两级危险工具清单

信任边界：每个频道、每个用户

核心洞察：安全模型的复杂度与使用场景直接相关。Open Code 面对的是"一个开发者坐在终端前"，所以交互式确认足够。 OpenClaw 面对的是"20 个平台上的不确定数量用户同时操作"，所以必须用声明式策略 + 分级审批 + owner 隔离的多层防护。 安全设计永远不是"够不够"，而是"场景要求什么"。

推荐阅读顺序

1️⃣先读 Agent 循环

3️⃣然后读多频道路由

resolve-route.ts + session.ts。这是 OpenClaw 独有的模块，理解消息如何从 20+ 平台路由到正确的 Agent 和 Session。

4️⃣最后读记忆和 Skills

compact.ts + memory/ + skills/。这些是"锦上添花"的模块，理解了前三步后，这些会很容易看懂。
