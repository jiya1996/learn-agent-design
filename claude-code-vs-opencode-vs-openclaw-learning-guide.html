Agent Loop — 核心循环的三种实现
从源码看：同一个本质，三种工程权衡
Agent Loop 的本质永远是 while stop_reason == "tool_use"。但在生产中，这个循环要处理流式输出、取消、权限阻塞、并发控制、错误恢复等真实世界的复杂性。三个产品的实现路径截然不同。

Claude Code 的实现（TypeScript，源自 cli.js 逆向 + SDK 类型定义）

🟣 CLAUDE CODE 层级式 Agent 系统

Claude Code 不是单一循环，而是一个层级式 Agent 体系：

TypeScript (架构伪码，基于 sdk-tools.d.ts 还原)
// Claude Code 的核心 Agent 循环
async function agentLoop(messages, tools, hooks) {
  while (true) {
    // 1. 流式调用 LLM
    const stream = await claude.messages.stream({
      model, system: buildSystemPrompt(hooks),
      messages, tools, max_tokens
    });

    // 2. 实时处理流式事件（逐 token 显示）
    const response = await processStream(stream);
    messages.push({ role: "assistant", content: response.content });

    // 3. 判断是否结束
    if (response.stop_reason !== "tool_use") return;

    // 4. 对每个工具调用：Hook → 权限 → 执行 → Hook
    const results = [];
    for (const toolCall of response.toolCalls) {
      await hooks.trigger("PreToolUse", toolCall);   // Hook 可以拦截
      const permitted = await checkPermission(toolCall); // 阻塞等待用户
      if (!permitted) { results.push(errorResult); continue; }

      const output = await executeTool(toolCall);       // 分发执行
      await hooks.trigger("PostToolUse", toolCall, output);
      results.push({ tool_use_id: toolCall.id, content: output });
    }

    // 5. 工具结果作为 user 消息喂回
    messages.push({ role: "user", content: results });
  }
}

独特设计：

  • Hook 系统包裹每次工具调用 — PreToolUse / PostToolUse 让用户的 Python/Shell 脚本拦截和修改工具行为
  • Sub-Agent 工具 — 循环中的 Agent 工具会递归创建新的 Agent Loop 实例，拥有独立 messages[]
  • Plan Mode — 特殊模式下 Agent 只生成计划不执行，用户审批后切换到执行模式
  • Context Auto-Compact — 接近 token 上限时自动压缩历史，通过 PreCompact Hook 通知

OpenCode 的实现（Go，完全开源）

🔵 OPENCODE agent.go:processGeneration()
Go (实际源码简化)
func (a *agent) processGeneration(ctx context.Context, sessionID string) {
    // 加载消息历史（支持 Summary 续接）
    msgs := a.messages.ListBySession(ctx, sessionID)
    if session.SummaryMessageID != "" {
        msgs = truncateFromSummary(msgs)  // 从摘要点截断
    }

    for {
        select {
        case <-ctx.Done(): return  // 取消支持
        default:
        }

        // 1. 流式调用 Provider（多 Provider 抽象）
        events := a.provider.StreamResponse(ctx, msgs, a.tools)
        assistantMsg := streamAndHandleEvents(events)  // 实时处理

        // 2. 检查是否需要继续
        if assistantMsg.FinishReason() != FinishReasonToolUse {
            a.messages.Save(ctx, assistantMsg)
            return
        }

        // 3. 顺序执行工具（带权限阻塞）
        toolResults := make([]ToolResult, len(assistantMsg.ToolCalls()))
        for i, tc := range assistantMsg.ToolCalls() {
            tool := findTool(a.tools, tc.Name)  // O(N) 线性搜索
            if !a.permissions.Request(tc) {   // 阻塞！等用户批准
                cancelRemaining(toolResults[i:])
                break
            }
            result, _ := tool.Run(ctx, tc)
            toolResults[i] = result
        }

        // 4. 追加到历史，继续循环
        msgs = append(msgs, assistantMsg, toolResultsMsg)
    }
}

独特设计：

  • Context 取消传播 — Go 的 context.Context 原生支持取消链，sync.Map 追踪每个 session 的 cancel func
  • Permission 通道阻塞 — make(chan bool, 1) + PubSub，TUI 弹窗响应后通过 channel 解除阻塞
  • Summary 续接 — SummaryMessageID 字段标记摘要位置，下次加载时从该点截断历史
  • LSP 诊断等待 — 编辑文件后等待语言服务器返回诊断信息，作为 metadata 附加到工具结果

OpenClaw 的实现（TypeScript，事件驱动）

🟠 OPENCLAW agent-command.ts (1333 行)

async function agentCommandInternal(opts: AgentCommandOpts) {
  // 1. 加载 JSONL 会话 + Bootstrap 文件
  const session = loadSessionTranscript(opts.sessionKey);
  const bootstrap = await applyBootstrapHookOverrides({
    files: [AGENTS_MD, SOUL_MD, IDENTITY_MD, USER_MD],  // 插件可修改！
  });

  // 2. 解析模型（支持 fallback chain）
  const model = resolveModelWithFallback(opts.modelOverride, cfg);

  // 3. 通过 SessionActorQueue 保证 per-session 串行
  await sessionActorQueue.run(opts.sessionKey, async () => {
    // 4. 选择运行时：嵌入式 Pi 或 ACP 子进程
    if (useAcp) {
      await spawnAcpDirect(opts);  // 独立进程
    } else {
      await runEmbeddedPiAgent({   // 嵌入式运行
        session, bootstrap, model, tools,
        onEvent: (event) => {
          // 5. 实时流式事件 → WebSocket → 客户端
          gateway.emit("agent:stream", event);
        }
      });
    }
  });
}

独特设计：

  • SessionActorQueue — 每个 session 同一时间只能跑一个请求，防止 JSONL 写入竞态
  • Lane 并发控制 — Main/Cron/Subagent 三种 lane 有独立并发上限（如 main:8, cron:1, subagent:10）
  • 双运行时 — 嵌入式 Pi（进程内）vs ACP 子进程（隔离），可按需切换
  • Bootstrap Hook — 插件在 Agent 看到 AGENTS.md/SOUL.md 之前就能修改内容
  • Model Fallback Chain — 主模型失败自动切换备选模型
关键差异：工具执行方式
Claude Code 和 OpenCode 都是顺序执行工具（一个接一个），OpenClaw 的 Pi 运行时支持并行执行多个工具。这影响了延迟和复杂度的权衡。

Permission System — 权限模型的三种哲学
从"完全信任"到"逐项审批"的权衡
权限系统是 Agent 产品最关键的信任边界。核心矛盾：自主性越高体验越好，但风险也越大。三个产品做出了完全不同的选择。

Claude Code 的 Deferred Tools 机制

🎯 按需加载工具定义——节省 System Prompt 空间

Claude Code 有 40+ 工具（内置 + MCP），如果全部塞进 system prompt 会浪费大量 token。解决方案：

机制说明
// 1. 启动时只加载核心工具的完整 schema
完整加载: Read, Edit, Write, Bash, Glob, Grep, Agent

// 2. 其他工具只在 <available-deferred-tools> 中列出名称
延迟加载: WebFetch, WebSearch, mcp__playwright__*, mcp__chrome__*, ...

// 3. Agent 需要某工具时，调用 ToolSearch 获取完整 schema
ToolSearch({ query: "select:WebFetch,WebSearch" })
→ 返回完整的 JSONSchema 定义
→ 工具变为可调用状态

这个机制让 Claude Code 在保持工具丰富性的同时，避免了 system prompt 膨胀。OpenCode 和 OpenClaw 都没有此机制。

OpenCode 的 PubSub 架构（源码级）

▶ Go 泛型 PubSub Broker 实现细节
Go (internal/pubsub/)
type Broker[T any] struct {
    subs map[chan Event[T]]struct{}
    mu   sync.RWMutex
}

// Subscribe：创建 buffered channel，绑定到 context 生命周期
func (b *Broker[T]) Subscribe(ctx context.Context) chan Event[T] {
    ch := make(chan Event[T], 64)  // buffer=64 防抖
    b.subs[ch] = struct{}{}
    go func() {
        <-ctx.Done()       // context 取消时自动清理
        b.mu.Lock()
        delete(b.subs, ch)
        close(ch)
        b.mu.Unlock()
    }()
    return ch
}

// Publish：非阻塞 fan-out（满则丢弃）
func (b *Broker[T]) Publish(typ EventType, payload T) {
    b.mu.RLock()
    snapshot := maps.Clone(b.subs)
    b.mu.RUnlock()
    for ch := range snapshot {
        select {
        case ch <- Event[T]{typ, payload}:
        default:  // 满了就丢！不阻塞发布者
        }
    }
}

设计要点：非阻塞发布确保 Agent 循环不被慢消费者拖住；Context 自动清理防止泄漏；泛型让同一个 Broker 服务于 Agent/Session/Message/Permission 等所有事件类型。

各产品优劣势详解

🟣 Claude Code 优劣势

优势 — 解决了什么问题

  • 最完整的 Agent 体验 — 从单 Agent 到多 Agent Teams，从 Plan Mode 到 Worktree 隔离，生态最成熟
  • Hook + MCP 双轨扩展 — 行为拦截和工具扩展分离，各自清晰
  • Memory 系统 — 唯一实现了结构化跨会话记忆的产品（user/feedback/project/reference 四种类型）
  • Deferred Tools — 按需加载工具定义，节省 system prompt 空间，独家机制
  • Plan Mode — "先说再做"的安全模式，让用户在执行前审查计划
  • Skill 系统 — Markdown 定义可复用知识模块，支持项目级/全局级/内置三层

劣势 — 留下了什么问题

  • 模型锁定 — 只能用 Claude，无法切换到 GPT-4/Gemini（最大限制）
  • 闭源 — 12.1MB 编译后 cli.js，无法审计安全性或自定义核心逻辑
  • 成本 — Opus 模型费用高昂，快速模式 6x 定价
  • 单终端 — 只有 CLI + IDE 插件，不支持 Web/Mobile/消息平台
  • 无离线模式 — 完全依赖 Anthropic API
🔵 OpenCode 优劣势

优势

  • 完全开源 — Go 源码可审计、可 fork、可定制
  • 多 Provider 支持 — 12+ Provider（Anthropic/OpenAI/Gemini/Bedrock/Copilot/Azure/...），模型自由切换
  • LSP 集成 — 唯一集成语言服务器的产品，编辑后自动获取诊断
  • Bubble Tea TUI — 精美的终端 UI，接近 IDE 体验
  • SQLite 持久化 — 结构化数据库存储（比 JSONL 更可靠、可查询）
  • Prompt Cache — 对 Anthropic API 自动应用缓存标记，降低成本

劣势

  • 无子代理 / 多 Agent — 单 Agent 架构，无法并行处理复杂任务
  • MCP 无刷新 — 启动时加载一次，运行中新增的 MCP 工具无法发现
  • 工具分发 O(N) — 线性搜索而非哈希查找（工具多时性能下降）
  • 无记忆系统 — 没有跨会话记忆，每次重启从零开始
  • 已归档 — 项目已停止维护，团队转向 Crush
  • 无扩展性 — 除 MCP 外没有 Hook/Plugin 机制
🟠 OpenClaw 优劣势

优势

  • 多渠道霸主 — 20+ 消息平台（WhatsApp/Telegram/Slack/Discord/Signal/iMessage/...）
  • Gateway 架构 — WebSocket RPC 控制面，支持任意客户端接入（CLI/Web/Mobile）
  • 插件系统最灵活 — ChannelPlugin + AgentTool + Hook 三层扩展
  • Lane 并发模型 — Main/Cron/Subagent 独立并发池，精细资源控制
  • Bootstrap Hook — 插件在 Agent 看到配置文件前就能修改内容，最强的注入控制
  • 语音 + Canvas — 超越文本的交互形态

劣势

  • 开发复杂度极高 — agent-command.ts 1333 行、server.impl.ts 56KB，入门门槛高
  • 单用户模型 — 信任模型简单，不适合多租户场景
  • 无结构化记忆 — 依赖 Bootstrap 文件而非分类记忆系统
  • Code 场景弱 — 不专注编程，缺少 LSP/编辑器集成
  • 部署门槛高 — 需要 Node.js 22+、Gateway 守护进程、各渠道 API 配置

Agent 设计指南 — 如果你要设计自己的 Agent
从三个产品的经验中提炼的设计决策框架
设计 Agent 不是选择"最好的架构"，而是在具体约束下做出正确的权衡。以下决策框架帮你在每个关键节点做出选择。

决策 1：Agent Loop 的复杂度

你需要多复杂的循环？

场景推荐方案参考实现
快速原型 / 学习30 行 while 循环learn-claude-code s01
单用户 CLI 产品+ 流式 + 取消 + 权限阻塞OpenCode agent.go
多用户 / 多渠道服务+ 并发控制 + Lane + Actor QueueOpenClaw agent-command.ts
企业级 + 多 Agent+ 子代理 + 任务板 + 通信协议Claude Code Agent Teams

原则：从最简单的循环开始，只在真正遇到问题时才加复杂度。learn-claude-code 的 12 节课完美展示了这个递进过程。

决策 2：工具系统设计

你的工具需要多大的扩展性？

选择适用场景权衡
硬编码工具集工具固定、场景单一最简单，但无法运行时扩展
Interface + 注册开发者需要自定义工具需要编译部署（OpenCode）
MCP 协议需要生态和第三方工具标准化但增加网络开销
Plugin 热加载需要运行时动态扩展最灵活但复杂度最高（OpenClaw）

原则：先硬编码核心工具（bash/read/write/edit/search），再用 MCP 开放扩展。这是 Claude Code 验证过的路径。

决策 3：权限模型

你的用户愿意被打断多少次？

模型适用场景信任度
全部自动批准受信环境、CI/CD高信任
黑名单 + 白名单快速原型中等信任
交互式确认CLI / IDE 场景低信任、高安全
Plan → Approve → Execute高风险操作最低信任、最高安全

原则：读操作宽松，写操作严格，不可逆操作必须确认。Claude Code 的三级分类（allow/ask/deny）是最佳实践。

决策 4：上下文管理

你预期的对话有多长？

对话长度推荐策略
< 50 轮不需要压缩，直接使用
50-200 轮旧工具结果替换为占位符（Micro Compact）
200+ 轮自动摘要 + 截断重置（Auto Compact）
跨会话Memory 系统 + Bootstrap 文件

原则：层层递进，按需触发。Deferred Tools 机制（按需加载工具定义）是一个经常被忽视但非常有效的优化。

决策 5：持久化选择

你需要什么级别的持久化？

需求推荐方案参考
对话历史JSONL（简单追加）或 SQLite（可查询）OpenClaw / OpenCode
跨会话记忆Markdown 文件 + 索引文件Claude Code Memory
团队协作状态JSON 文件 + 依赖图Claude Code Tasks
配置管理分层合并（global → project → local）三个产品都采用

决策 6：多 Agent 架构

你需要多少个 Agent 协作？

规模推荐架构关键组件
单 Agent简单循环即可Agent Loop
主 + 子独立 messages[] 的子代理+ Sub-Agent 工具
小团队 (2-5)共享任务板 + 点对点消息+ Task Board + SendMessage
大团队 (5+)Lane 并发 + Worktree 隔离+ SessionActorQueue + Git Worktree

原则：不要过早引入多 Agent。大多数任务单 Agent + 好的工具集就够了。只在任务确实可并行、且需要角色专业化时才升级。

推荐技术栈选择

你的目标推荐学习路径
理解 Agent 原理learn-claude-code (Python, 12 节课从 0 到 1)
构建开发者 CLI 工具参考 OpenCode (Go) 或 Claude Code 的架构模式
